Tsundere Protect

Tsundere Protect is the protection pipeline for Node.js runtime output. The current version includes the first working foundation: protected builds, source map stripping, string encoding, runtime metadata, and build fingerprints.

Current Commands

tsundere build --protect standard
tsundere build --protect advanced
tsundere build --protect maximum
tsundere build --protect advanced --seed my-build
tsundere fingerprint inspect
tsundere fingerprint inspect .tsundere/runtime-build/main.js

Use `--seed` when you need deterministic fingerprints for repeated builds. Omit it, or use `--seed auto`, when every build should look different.

Standard

Whitespace minification, source map stripping, string encoding, and build fingerprint metadata. Low performance impact.

Advanced

Standard features plus runtime integrity checks and generated guard code. More advanced control-flow work is planned next.

Maximum

Advanced features plus an extra runtime wrapper layer. Runtime virtualization and layered decryptors are planned future work.

Protection Layers

Runtime integrity: Advanced and maximum builds insert runtime checks so protected output can fail fast when helper code is obviously damaged.

String encoding: Static strings are encoded with a build-specific helper. Template strings with interpolation are left alone for compatibility.

Build fingerprints: Protected files get unique build IDs and metadata in `.tsundere/runtime-build/tsundere-protect.json`.

Seeded builds: `--seed my-build` produces deterministic fingerprints. `--seed auto` or no seed produces randomized build IDs.

Future layers: Control-flow flattening, asset protection, stronger string encryption, environment protection, and anti-debugging are roadmap items.

Fingerprinting

Protected builds receive unique IDs that can be inspected from metadata or from a protected output file.

tsundere fingerprint inspect app.js
