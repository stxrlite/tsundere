Transition to Yuri

Yuri is familiar if you know JavaScript, but Tsundere is not meant to be a plain one-to-one copy. The Discord workflow uses command discovery, typed wrapper APIs, stricter diagnostics, and project layouts that remove repeated bot boilerplate.

From JavaScript

JavaScript

client.on("interactionCreate", async (i) => {
  if (!i.isChatInputCommand()) return
  if (i.commandName === "ping") {
    await i.reply("pong")
  }
})

Yuri

client.on("interactionCreate", async (interaction) => {
  if interaction.isCommand("ping") {
    await interaction.reply("pong")
  }
})

The big change is not just syntax. Tsundere tries to infer Discord intent needs, validate Discord limits, and suggest wrapper methods directly in the editor.

From Python

Python Bot Style

@bot.event
async def on_ready():
    print("online")

@bot.command()
async def ping(ctx):
    await ctx.reply("pong")

Yuri Bot Style

client.once("ready", () => {
  log("online")
})

Slash.command("ping")
  .description("Reply with pong")
  .register(client)

Python decorators usually hide registration inside the framework. Tsundere keeps registration explicit or discoverable through command files, so you can still understand what gets sent to Discord.

Core Differences

ConceptHow Yuri Handles It
importsUse normal package imports, including npm packages and `@tsundere/discord`.
eventsUse `client.on` and `client.once`, with Discord-aware autocomplete and intent diagnostics.
commandsUse `Slash.command` or command files in `src/commands` for discovery.
repliesUse typed interaction helpers like `reply`, `deferReply`, `editReply`, and `followUp`.
buildersUse Tsundere builders for embeds, buttons, rows, modals, and selects.
runtimeUse `tsundere dev`, `tsundere build`, and `tsundere start` instead of manually running generated files.
